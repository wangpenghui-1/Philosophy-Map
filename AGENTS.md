# 开发协作与 Git 发布流程

本项目将“实现与验证”和“版本发布”分为两个明确阶段。除非用户在当前对话中明确授权，代理不得自行改变 Git 的发布状态。

## 职责边界

- 代理可在已明确的任务范围内自主阅读代码、实现修改、运行必要的本地验证，并报告结果。
- 代理可使用只读 Git 操作（如 `status`、`diff`、`log`、`show`）了解范围与基线。
- 用户负责决定何时形成版本、何时向远端发布、是否创建或合并 PR，以及是否部署。
- 代理不得索取、显示、保存或转发 GitHub 密码、个人访问令牌、cookie、会话文件或其他原始凭据。

## 阶段检查点

每个可交付阶段完成后，代理必须先报告：

1. 已完成的修改及影响范围；
2. 已执行的验证及结果；
3. 当前工作区中待提交的文件，以及任何既有或无关的改动；
4. 建议的下一步 Git 操作。

随后必须等待用户的明确同意，才可执行对应的 Git 写操作。确认应与操作粒度相匹配：

- 创建或切换分支：需要明确同意；
- 暂存或本地提交：需要阶段完成后的明确同意；
- 推送远端：需要明确同意；
- 创建 PR：需要明确同意，并明确草稿或准备审阅状态；
- 合并、打标签、发布或部署：始终需要单独的明确同意。

“继续”“处理一下”等笼统表达不构成 Git 发布授权；“提交”“推送”“创建 PR”等明确指令只授权所述操作，不自动扩大为其他发布操作。

## 范围保护

- 绝不使用 `git add -A` 或类似方式默认纳入整个工作区。
- 工作区混有既有或无关改动时，先区分基线与本任务差异；无法安全隔离时，停止并说明原因。
- 推送前再次核对分支、上游、待推送提交和未提交改动。
- 未获得授权时，保持修改与验证结果在本地工作区，不创建提交、不推送、不创建 PR。

## 分支约定

- 从默认分支开始的新工作优先使用 `codex/<简短任务名>`；如用户指定分支或当前分支已有明确任务，则遵从用户指定。
- 不得因完成代码而自动创建、切换、删除或重置分支。
